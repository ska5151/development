# Everything Claude Code (ECC) — 에이전트 지침

소프트웨어 개발을 위해 **54개의 전문 에이전트**, **205개의 스킬**, **69개의 커맨드**, **자동화된 훅 워크플로**를 제공하는 **프로덕션 수준의 AI 코딩 플러그인**입니다.

**버전:** 2.0.0-rc.1

## 핵심 원칙

1. **에이전트 우선** — 도메인 작업은 전문 에이전트에 위임합니다.
2. **테스트 주도** — 구현 전에 테스트를 작성하며, 커버리지 **80% 이상**을 요구합니다.
3. **보안 우선** — 보안을 타협하지 않으며, 모든 입력을 검증합니다.
4. **불변성** — 항상 새 객체를 만들고, 기존 객체를 직접 변경하지 않습니다.
5. **실행 전 계획** — 복잡한 기능은 코드 작성 전에 계획합니다.

## 에이전트 오케스트레이션

사용자가 명시적으로 요청하지 않아도 적극적으로 에이전트를 사용합니다.

- 복잡한 기능 요청 → **planner**
- 방금 작성·수정한 코드 → **code-reviewer**
- 버그 수정 또는 신규 기능 → **tdd-guide**
- 아키텍처 결정 → **architect**
- 보안에 민감한 코드 → **security-reviewer**
- 자율 루프 / 루프 모니터링 → **loop-operator**
- 하네스 설정의 안정성과 비용 → **harness-optimizer**

서로 독립인 작업은 **병렬 실행**합니다. 여러 에이전트를 동시에 기동합니다.

## 보안 가이드라인

**커밋 전 반드시:**

- 하드코딩된 비밀(API 키, 비밀번호, 토큰) 없음
- 모든 사용자 입력 검증
- SQL 인젝션 방지(파라미터화된 쿼리)
- XSS 방지(HTML 정제)
- CSRF 보호 활성화
- 인증·인가 검증
- 모든 엔드포인트에 속도 제한
- 오류 메시지에 민감 정보 노출 없음

**비밀 관리:** 비밀을 **절대** 하드코딩하지 않습니다. 환경 변수 또는 시크릿 매니저를 사용합니다. 시작 시 필요한 비밀을 검증합니다. 노출된 비밀은 즉시 로테이션합니다.

**보안 이슈 발견 시:** **중단** → **security-reviewer** 에이전트 사용 → **CRITICAL** 이슈 수정 → 노출된 비밀 로테이션 → 유사 패턴 전체 코드베이스 검토

## 코딩 스타일

**불변성(필수):** 항상 새 객체를 만들고 직접 변경하지 않습니다. 변경이 반영된 새 복사본을 반환합니다.

**파일 구성:** 큰 파일 소수보다 작은 파일 다수를 선호합니다. 보통 200~400줄, 최대 800줄. 타입별이 아니라 **기능·도메인** 단위로 구성합니다. 응집도는 높이고 결합도는 낮춥니다.

**오류 처리:** 모든 계층에서 오류를 처리합니다. UI 코드에는 사용자 친화적 메시지를, 서버에는 상세 컨텍스트를 로깅합니다. 오류를 조용히 삼키지 않습니다.

**입력 검증:** 시스템 경계에서 모든 사용자 입력을 검증합니다. 스키마 기반 검증을 사용합니다. 명확한 메시지로 빠르게 실패합니다. 외부 데이터를 신뢰하지 않습니다.

**코드 품질 체크리스트:**

- 함수는 작게(50줄 미만), 파일은 한 주제에 집중(800줄 미만)
- 깊은 중첩 없음(4단계 초과 금지)
- 적절한 오류 처리, 하드코딩된 값 없음
- 읽기 쉽고 이름이 잘 드러나는 식별자

## 테스트 요구사항

**최소 커버리지: 80%**

테스트 유형(모두 필요):

1. **단위 테스트** — 개별 함수, 유틸, 컴포넌트
2. **통합 테스트** — API 엔드포인트, DB 작업
3. **E2E 테스트** — 핵심 사용자 플로우

**TDD 워크플로(필수):**

1. 테스트 먼저 작성(RED) — 테스트는 **실패**해야 함
2. 최소 구현(GREEN) — 테스트는 **통과**해야 함
3. 리팩터(IMPROVE) — 커버리지 80% 이상 확인

실패 시: 테스트 격리 확인 → 목(mock) 검증 → 구현 수정(테스트가 잘못된 경우가 아니면 테스트를 고치지 않음)

## 개발 워크플로

1. **계획** — planner 에이전트 사용, 의존성·리스크 파악, 단계로 분해
2. **TDD** — tdd-guide 에이전트 사용, 테스트 우선, 구현, 리팩터
3. **리뷰** — code-reviewer 에이전트를 즉시 사용, CRITICAL/HIGH 이슈 처리
4. **지식을 적절한 위치에 기록**
   - 개인 디버깅 메모, 선호, 임시 맥락 → 자동 메모리
   - 팀·프로젝트 지식(아키텍처 결정, API 변경, 런북) → 프로젝트 기존 문서 구조
   - 현재 작업에서 이미 관련 문서나 코드 주석이 나왔다면 동일 정보를 다른 곳에 중복하지 않음
   - 문서 위치가 불분명하면 새 최상위 파일을 만들기 전에 확인 요청
5. **커밋** — Conventional Commits 형식, PR 요약은 충분히 작성

## 워크플로 표면 정책

- `skills/`가 **정식(canonical)** 워크플로 표면입니다.
- 새 워크플로 기여는 우선 `skills/`에 반영합니다.
- `commands/`는 레거시 슬래시 진입 호환용이며, 마이그레이션 또는 크로스 하네스 호환을 위한 시밈이 여전히 필요할 때만 추가·갱신합니다.

## Git 워크플로

**커밋 형식:** `<type>: <설명>` — 타입: feat, fix, refactor, docs, test, chore, perf, ci

**PR 워크플로:** 전체 커밋 이력 분석 → 요약 초안 → 테스트 계획 포함 → `-u` 플래그로 푸시

## 아키텍처 패턴

**API 응답 형식:** 성공 여부, 데이터 페이로드, 오류 메시지, 페이지네이션 메타데이터를 담은 일관된 봉투(envelope) 형태

**리포지토리 패턴:** 데이터 접근을 표준 인터페이스(findAll, findById, create, update, delete) 뒤에 캡슐화합니다. 비즈니스 로직은 저장 메커니즘이 아닌 추상 인터페이스에 의존합니다.

**스켈레톤 프로젝트:** 검증된 템플릿을 찾고, 병렬 에이전트(보안, 확장성, 적합성)로 평가한 뒤 최적 후보를 클론하고, 검증된 구조 안에서 반복합니다.

## 성능

**컨텍스트 관리:** 대규모 리팩터링·다중 파일 기능에서는 컨텍스트 창 **마지막 20%** 사용을 피합니다. 민감도가 낮은 작업(단일 편집, 문서, 단순 수정)은 더 높은 사용률을 허용합니다.

**빌드 문제 해결:** build-error-resolver 에이전트 사용 → 오류 분석 → 점진적 수정 → 수정마다 검증

## 프로젝트 구조

```
agents/          — 54개 전문 서브에이전트
skills/          — 205개 워크플로 스킬 및 도메인 지식
commands/        — 69개 슬래시 커맨드
hooks/           — 트리거 기반 자동화
rules/           — 항상 따를 지침(공통 + 언어별)
scripts/         — 크로스 플랫폼 Node.js 유틸
mcp-configs/     — 14개 MCP 서버 설정
tests/           — 테스트 스위트
```

`commands/`는 호환을 위해 저장소에 남아 있으나, 장기 방향은 **스킬 우선(skills-first)** 입니다.

## 성공 지표

- 모든 테스트 통과, 커버리지 80% 이상
- 보안 취약점 없음
- 코드가 읽기 쉽고 유지보수 가능함
- 성능이 허용 범위 내
- 사용자 요구사항 충족
